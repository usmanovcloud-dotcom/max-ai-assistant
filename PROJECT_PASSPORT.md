## 1. Кратко о проекте

`max-ai-assistant` — персональный AI-ассистент для отдельного аккаунта MAX. Он принимает только личные входящие сообщения привязанного владельца, обращается к OpenAI-совместимому Responses API и возвращает ответ в MAX. Проект также содержит локальную web-панель, локальное хранение состояния в SQLite, безопасное управление секретами, статистику, аудит, backup и отключаемую интеграцию с n8n.

Основной профиль LLM — официальный OpenAI. OpenRouter поддерживается как дополнительный провайдер. Основной транспорт MAX — `maxapi-python`/PyMax `2.3.1`.

## 2. Источники правды и их приоритет

При расхождении сведений использовать следующий порядок:

1. Фактически запущенный runtime и его данные — только после отдельной живой проверки.
2. Текущий код и конфигурация в рабочем дереве.
3. Unit-тесты, воспроизводящие ожидаемое поведение.
4. Git-коммиты и теги — для истории уже зафиксированных версий.
5. Этот паспорт — для общего контекста, статуса и списка известных расхождений.
6. `README.md`, `NAS_ONE_YAML.md` и инструкции миграции — эксплуатационная документация, которая может временно отставать от кода.

Важно: прохождение локальных тестов не доказывает успешную публикацию образа, обновление NAS, работоспособность MAX или доступность внешнего LLM API.

## 5. Текущая архитектура

- `app/core.py` — owner gate, дедупликация, история, вызов responder, chunked send и безопасный retry.
- `app/storage.py` — SQLite: владелец, сообщения, диалоги, настройки, лимиты, статистика и аудит.
- `app/pymax_transport.py` — MAX/PyMax, QR/2FA, reconnect, фильтрация чатов и преобразование вложений.
- `app/providers/openai_compatible.py` — OpenAI/OpenRouter Responses API, лимиты, retry и безопасная диагностика.
- `app/runtime.py`, `app/supervisor.py` — сборка runtime и управление AI-процессом.
- `app/web.py`, `app/dashboard.py`, `app/web_static/` — локальная панель и API управления.
- `app/n8n.py` — отключаемая отправка событий в webhook; ошибка n8n не блокирует ответ владельцу.
- `docker-compose.yml` — локальный безопасный compose.
- `docker-compose.nas.yml` — вариант для Synology с готовым GHCR-образом.
- `.github/workflows/docker-publish.yml` — тестирование и публикация `linux/amd64` образа.
- `tests/` — локальная спецификация ключевого поведения.

## 6. Инварианты безопасности

- Принимаются только входящие личные сообщения точного сочетания owner sender ID + owner chat ID.
- Telemetry PyMax должна оставаться принудительно выключенной.
- API-ключи, MAX session, 2FA, login URL, подписанные URL вложений и текст сообщений не должны попадать в Git или обычные логи.
- Claim-код хранится только как hash, ограничен сроком и одноразовым использованием.
- Responses API вызывается с `store=false`; история управляется локально.
- Вложения передаются только по проверенным HTTPS URL и только в текущем запросе.
- Панель NAS допустима только в доверенной LAN. Порт `8765` нельзя публиковать в интернет без отдельной аутентификации и TLS.
- Backup не должен включать MAX session и секреты провайдеров.
- Любое изменение send/retry логики должно сохранять идемпотентность и не вызывать LLM повторно после уже сохранённого ответа.

## 8. Известные расхождения и риски

1. Обработка вложений и аудио проверена mock/unit-тестами, но не подтверждена реальным событием PyMax и реальными OpenAI запросами.
2. Аудио распознаётся только при выбранном официальном OpenAI-провайдере; OpenRouter не используется для `/audio/transcriptions`.
3. Новый NAS storage-init зависит от поведения Docker Compose/Container Manager для `depends_on.condition: service_completed_successfully`; требуется проверка на целевой DSM.
4. UID/GID `1032:100` согласован во всех NAS-файлах как рабочая база пользователя, но фактические права bind mounts нужно подтвердить на NAS после миграции.
5. Docker отсутствует на локальной машине, поэтому compose/build не проверены локально.

## 9. Ближайший безопасный порядок работ

1. Проверить Docker build/compose в CI или на машине с Docker.
2. Сделать резервную копию старых volumes и провести контролируемую NAS-миграцию с проверкой mounts, UID/GID, SQLite, настроек, ключей и MAX session.
3. Провести живой smoke test текста, PDF, Word, Excel, PowerPoint, изображения, голосового MAX и обычного аудиофайла.
4. Проверить новый QR, 2FA, копирование `/claim` и сообщение `Готовлю ответ…` в реальном MAX.
5. После живой проверки отправить `main`, открыть PR при необходимости, создать tag `v0.4.0` и дождаться GHCR-образа.
6. Наблюдать NAS-runtime 24–48 часов перед удалением старых именованных volumes.

После каждого содержательного этапа обновлять:

- дату и активную ветку/HEAD;
- новый пункт хронологии с датой, commit/PR/tag и фактическим результатом;
- раздел текущего рабочего состояния;
- число и результат тестов;

- новые риски и закрытые расхождения;

Нельзя записывать в паспорт как завершённое то, что только подготовлено в файлах. Формулировки должны различать `реализовано локально`, `прошло unit-тесты`, `опубликовано`, `развёрнуто на NAS` и `проверено в живом runtime`.
